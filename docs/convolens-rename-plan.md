### Non-mechanical: localStorage migration

`packages/contexts/src/auth-context.tsx` reads and writes `whatssummarize_user`
and `whatssummarize_session` to `localStorage`. A naive rename will sign every
existing user out on next deploy. Add a one-shot migration shim inside the
auth-context init effect:

```ts
const OLD_USER = 'whatssummarize_user'
const OLD_SESSION = 'whatssummarize_session'
const USER_STORAGE_KEY = 'convolens_user'
const SESSION_STORAGE_KEY = 'convolens_session'

// Migrate legacy keys, run-once
if (typeof window !== 'undefined') {
  for (const [oldK, newK] of [[OLD_USER, USER_STORAGE_KEY], [OLD_SESSION, SESSION_STORAGE_KEY]]) {
    const existing = localStorage.getItem(newK)
    const legacy = localStorage.getItem(oldK)
    if (legacy && !existing) localStorage.setItem(newK, legacy)
    if (legacy) localStorage.removeItem(oldK)
  }
}
```

Drop the shim 30 days after deploy or when telemetry shows zero legacy reads.

### Smoke-test checklist

- [ ] Existing user with `whatssummarize_session` cookie/localStorage stays signed in across the deploy
- [ ] Page title reads "ConvoLens" in browser tab
- [ ] No `WhatsSummarize` strings in rendered HTML (`curl localhost:3000 | grep -i whatssummarize` → 0)

## PR 3 — Chrome extension brand sweep

**Branch:** `chore/convolens-chrome-extension`
**Estimated files:** ~10 source files, 1 manifest, 2 HTML
**Risk:** **HIGH** — **flag for user**: changing `host_permissions` may force Chrome Web Store re-review

### Files / scope

- `apps/chrome-extension/manifest.json` — `name`, `description`, `host_permissions: ["https://api.whatssummarize.com/*"]`, `web_accessible_resources`
- `apps/chrome-extension/popup.html`, `apps/chrome-extension/options.html` — title + visible copy
- `apps/chrome-extension/src/content.ts` (~12 console.logs, DOM ID `whatssummarize-fab`)
- `apps/chrome-extension/src/background.ts` (~10 console.logs)
- `apps/chrome-extension/src/config.ts` / `config.js`
- `apps/chrome-extension/README.md`

### Non-mechanical

- **Chrome Web Store implications**:
  - Changing `name` and `description` in `manifest.json` requires a new submission **but NOT a re-review** (metadata-only changes auto-approve in 24–48h).
  - Changing `host_permissions` to `https://api.convolens.com/*` **DOES** trigger re-review (typically 7–14 days). If the API domain is also moving, batch this with PR4.
  - **Recommendation:** Keep old host permission as an additional entry (`["https://api.whatssummarize.com/*", "https://api.convolens.com/*"]`) until the API DNS cutover is settled, then drop the legacy entry in a follow-up.
- DOM ID `whatssummarize-fab` is queried in two places — global rename, no semantic risk.
- The `package` script also produces `whatssummarize-extension.zip` — rename to `convolens-extension.zip`.

### Smoke-test checklist

- [ ] Load unpacked extension; click extension icon → popup shows "ConvoLens"
- [ ] Open WhatsApp Web → FAB element with id `convolens-fab` appears
- [ ] Manifest validator (`npx web-ext lint`) passes
- [ ] Background script reaches existing `api.whatssummarize.com` (until API moves)
